## Patch Application Fixes

### 1. `drivers/char/mspec.c` — Hunk #7 FAILED (vm_flags API change)

**Root cause:** Linux 6.3 converted `vma->vm_flags |=` to the new
`vm_flags_set()` helper function. The patch expected the old direct
assignment syntax as context, causing the hunk to fail.

**6.2 (patch expected):**
```c
vma->vm_flags |= VM_IO | VM_PFNMAP | VM_DONTEXPAND | VM_DONTDUMP;
if (vdata->type == MSPEC_UNCACHED)
```

**6.3 (actual):**
```c
vm_flags_set(vma, VM_IO | VM_PFNMAP | VM_DONTEXPAND | VM_DONTDUMP);
if (vdata->type == MSPEC_UNCACHED)
```

**Fix:** Manually applied the SN2 change to the correct line (269),
changing:
```c
if (vdata->type == MSPEC_UNCACHED)
```
to:
```c
if (vdata->type == MSPEC_FETCHOP || vdata->type == MSPEC_UNCACHED)
```

This adds MSPEC_FETCHOP (SN2 fetchop memory) to the noncached page
protection condition, which is required because fetchop memory uses
uncached MMIO-mapped pages on the SHUB.

## Upstream Changes in 6.3 (Non-Breaking)

These upstream changes between 6.2 and 6.3 caused offset adjustments
but did not require any code changes to the SN2 patch:

- **`arch/ia64/mm/contig.c`**: Refactored ZONE_DMA32 assignment to use
  a local `max_dma` variable. Our `#ifdef CONFIG_ZONE_DMA32` guard
  still applies correctly (wraps the ZONE_DMA32 reference).

- **`arch/ia64/mm/discontig.c`**: Same ZONE_DMA32 refactoring as
  contig.c. Our `#ifdef CONFIG_ZONE_DMA32` guard still applies.

- **`drivers/char/mspec.c`**: `vma->vm_flags |=` converted to
  `vm_flags_set()` helper (6.3 VMA flags API hardening). Only
  affected the context line, not the SN2-specific change itself.

- **`drivers/scsi/qla2xxx/qla_init.c`**: Additional abort handling
  code added (`qla_put_fw_resources`, `DSC_DELETED` state), wwn
  variable removed. Our `#include <asm/machvec.h>` at the top was
  unaffected.

- **`include/linux/pci_ids.h`**: New vendor IDs added (WANGXUN,
  SOLIDRUN). Our `PCI_DEVICE_ID_SGI_IOC4` addition unaffected.

## Build Fixes

### 1. `arch/ia64/sn/kernel/tiocx.c` — const device uevent callback

**Error:**
```
arch/ia64/sn/kernel/tiocx.c:147:19: error: initialization of
  'int (*)(const struct device *, struct kobj_uevent_env *)' from
  incompatible pointer type 'int (*)(struct device *, struct kobj_uevent_env *)'
```

**Cause:** Linux 6.3 changed the `bus_type.uevent` callback signature to
take `const struct device *` instead of `struct device *` (commit 5e33dc6
"driver core: make uevent callback const").

**Fix:** Add `const` qualifier to the `dev` parameter of `tiocx_uevent()`.
The function body (`return -ENODEV`) does not modify the device, so this
is a trivial signature-only change.
