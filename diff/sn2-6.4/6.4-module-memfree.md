# SN2 v6.3 → v6.4 Module Loading Fix

## Context

The v6.3 → v6.4 forward-port required one build fix (class_create API change
in snsc.c, documented in 6_4-fixes.md) and one runtime fix documented here.

An additional upstream bug (ARCH_SHF_SMALL sections mapped to MOD_INVALID)
broke module loading on all ia64 machines in v6.4-rc1. This was reported by
Frank Scheiner on the Debian ia64 mailing list and fixed by Song Liu in
commit `4d1f2cb26f8f` ("module: fix module load for ia64") before the v6.4
release. The fix is already present in released v6.4 and does not need to
be carried in the SN2 patch.

## Issue: VM_FLUSH_RESET_PERMS re-added by module_enable_x()

**File:** `arch/ia64/sn/kernel/setup.c`

### Background

Since v5.2, the SN2 patch includes a `module_alloc()` override that allocates
module memory without the `VM_FLUSH_RESET_PERMS` flag. That flag causes
`vfree()` to take an aggressive TLB flush path (`vm_reset_perms()` →
`_vm_unmap_aliases()` → `flush_tlb_kernel_range()`), which on SN2 triggers
SHUB PTC.GA broadcasts that interfere with concurrent PIO/MMIO traffic. The
override was sufficient through v6.3.

### What changed in v6.4

The module strict-RWX code in `kernel/module/strict_rwx.c` was refactored
as part of the module_memory conversion. A new shared helper
`module_set_memory()` was introduced:

```c
/* v6.4 kernel/module/strict_rwx.c */
static void module_set_memory(const struct module *mod, enum mod_mem_type type,
                              int (*set_memory)(unsigned long start, int num_pages))
{
    const struct module_memory *mod_mem = &mod->mem[type];

    set_vm_flush_reset_perms(mod_mem->base);  /* unconditionally sets the flag */
    set_memory((unsigned long)mod_mem->base, mod_mem->size >> PAGE_SHIFT);
}
```

`module_enable_x()` now calls this helper for all text sections:

```c
void module_enable_x(const struct module *mod)
{
    for_class_mod_mem_type(type, text)
        module_set_memory(mod, type, set_memory_x);
}
```

`module_enable_x()` is called unconditionally during module loading
(the comment in strict_rwx.c says: "keep module_enable_x() independent
of CONFIG_STRICT_MODULE_RWX because they are needed regardless").

In v6.3, `set_vm_flush_reset_perms()` was only called inside
`module_enable_ro()`, which exits early when `CONFIG_STRICT_MODULE_RWX`
is not set. ia64 does not select that config (its `set_memory_*` functions
are all no-ops), so the flag was never set and the `module_alloc()`
override was sufficient.

In v6.4, the flag is set on every module text section regardless of
config, bypassing the `module_alloc()` override.

### Crash sequence

1. `module_alloc()` — SN2 override allocates without `VM_FLUSH_RESET_PERMS` ✓
2. `module_enable_x()` → `module_set_memory()` → `set_vm_flush_reset_perms()`
   — re-adds `VM_FLUSH_RESET_PERMS` to every text section ✗
3. Module init completes, init sections freed via `do_free_init()` →
   `module_memfree()` → `vfree()`
4. `vfree()` sees `VM_FLUSH_RESET_PERMS` → `vm_reset_perms()` →
   `_vm_unmap_aliases()` → `flush_tlb_kernel_range()` →
   `sn2_global_tlb_purge()` → SHUB PTC.GA broadcast
5. PTC.GA collides with active PIO/MMIO from USB controller probing →
   SHUB returns corrupted data → intermittent kernel crash

### Symptoms

Same as the v5.2 VM_FLUSH_RESET_PERMS issue: intermittent crashes during
USB device probing with corrupted addresses, varying fault types across
boots (paging request, illegal operation, NaT consumption). The "Modules
linked in:" line shows USB and audio modules being loaded when the crash
occurs.

### Fix

Override `module_memfree()` (which is `__weak`) to clear
`VM_FLUSH_RESET_PERMS` before calling `vfree()`:

```c
void module_memfree(void *module_region)
{
    struct vm_struct *vm;

    if (!module_region)
        return;
    vm = find_vm_area(module_region);
    if (vm)
        vm->flags &= ~VM_FLUSH_RESET_PERMS;
    vfree(module_region);
}
```

This catches the flag regardless of how or when it gets added. The
`module_alloc()` override is still needed — it prevents the flag from
being set during allocation and uses `PAGE_KERNEL` instead of the default
`PAGE_KERNEL_EXEC`.

`find_vm_area()` is declared in `<linux/vmalloc.h>`, already included
in setup.c for the `module_alloc()` override.

### Why module_memfree and not other approaches

- `set_vm_flush_reset_perms()` is a static inline in a header — cannot
  be overridden per-arch
- `module_set_memory()` is static in strict_rwx.c — cannot be overridden
- `module_enable_x()` is not a weak symbol — cannot be overridden
- `module_memfree()` IS `__weak` — the designed override point for
  arch-specific module freeing

### Impact

SN2 only. The `module_memfree()` override is in `arch/ia64/sn/kernel/setup.c`,
which is only compiled when `CONFIG_IA64_SGI_SN2` is set. Other ia64
platforms (Integrity, etc.) are not affected by SHUB PTC.GA interference
and do not need this fix.
